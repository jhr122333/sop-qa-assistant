# SOP Q&A Assistant — 서비스 기획서

> 버전: v2.0 | 작성일: 2026-03-20 | 작성자: 서비스 기획팀

---

## 1. 문제 정의

### 1.1 현장 상황

제약·바이오 제조 현장에서는 GMP 준수를 위해 수십~수백 종의 SOP 문서를 운용한다. 현장 작업자가 절차를 확인해야 할 때 기존 프로세스는 다음 흐름을 거친다.

```
작업자 질문 발생
    → EDMS/공유드라이브에서 문서 검색 (5~15분)
    → 해당 SOP PDF 열어 섹션 탐색 (3~10분)
    → 내용 불명확 시 QA/선임에게 재질문 (대기 시간 불확실)
    → 답변 확보 후 작업 재개
```

**평균 답변 소요 시간: 20~40분 / 건** (내부 추정치)

### 1.2 핵심 문제점

| # | 문제 | 영향 |
|---|------|------|
| 1 | 문서가 많고 검색이 어렵다 | 올바른 SOP 섹션 도달까지 시간 낭비 |
| 2 | 담당자에게 직접 물어봐야 해결된다 | QA/선임 시간 잠식, 병목 발생 |
| 3 | 답변의 근거(출처)가 불명확하다 | 규정 해석 오류, 컴플라이언스 리스크 |
| 4 | 반복 질문이 누적되나 분석되지 않는다 | 운영 개선 기회 소실 |
| 5 | 챗봇 답변 품질이 검증되지 않는다 | 잘못된 절차 수행 가능성 |

## 2. 타겟 사용자

### 2.1 Primary — 제약·바이오 품질관리(QA/QC) 담당자

- **역할**: 배치 기록 검토, 환경 모니터링, OOS/Deviation 처리
- **Pain Point**: 절차 확인에 반복 시간 소모, 선임 의존도 높음
- **기대 효과**: 즉각적 답변으로 업무 속도 향상, 컴플라이언스 신뢰도 확보

### 2.2 Secondary — 신규 입사자·교육 담당자

- **역할**: GMP 절차 학습, 현장 교육 자료 준비
- **Pain Point**: 문서 범위가 넓어 학습 경로 불명확
- **기대 효과**: 질문 기반 학습, 교육 자료 준비 효율화

### 2.3 Tertiary — 관리자(QA Manager, 시스템 운영자)

- **역할**: 서비스 품질 모니터링, SOP 커버리지 관리
- **Pain Point**: 챗봇 답변 품질과 사용 패턴을 가시화할 수단 없음
- **기대 효과**: 대시보드로 실시간 품질 지표 확인, 개선 우선순위 파악

## 3. 경쟁·유사 서비스 분석

### 3.1 기존 사내 EDMS/문서 검색 시스템

| 항목 | 한계 |
|------|------|
| 검색 방식 | 파일명·태그 기반 키워드 검색 — 섹션 단위 검색 불가 |
| 답변 형태 | 문서 링크 반환 — 직접 읽어야 함 |
| 출처 연결 | 문서 단위 링크만 제공, 섹션 직접 연결 없음 |
| 피드백 루프 | 없음 |

### 3.2 범용 ChatGPT 활용

| 항목 | 한계 |
|------|------|
| SOP 특화 | 사내 SOP 미탑재 — 일반 GMP 지식만 제공 |
| 출처 추적 | 할루시네이션 위험, 출처 불명 |
| 컴플라이언스 | 사내 승인 문서 기반 답변 불가 |
| 사용 로그 | 없음 — 감사(Audit) 추적 불가 |

### 3.3 본 서비스의 차별점

```
SOP 특화 RAG
  → 승인된 사내 SOP 문서만을 지식 베이스로 사용
  → 답변마다 출처 섹션 명시 (Document ID + Section Title + Version)

피드백 기반 고도화
  → 👍/👎 피드백 수집 → 품질 지표 자동 집계
  → 부정 피드백 리뷰 큐 → 운영자 개선 루프

관리자 대시보드
  → 질문 패턴, 만족도, SOP 커버리지 시각화
  → 실데이터 기반 지속 개선 사이클
```

## 4. 핵심 기능 & 서비스 플로우

### 4.1 사용자 플로우

```
[사용자 질문 입력]
    ↓
[SOP 문서 검색 (BM25 + Field Boost + Phrase Boost)]
    ↓
[관련 섹션 추출 (상위 4개 Chunk)]
    ↓
[AI 답변 생성 (OpenAI GPT — JSON 구조화 출력)]
    ↓
[답변 + 출처(Document ID / Section / Version) 표시]
    ↓
[피드백 UI 표시 (👍 / 👎)]
    ↓
[피드백 저장 → feedback-log.json / localStorage]
```

### 4.2 관리자 플로우

```
[대시보드 접속]
    ↓
[KPI 확인: 질문 수, 답변 성공률, 부정 피드백]
    ↓
[응답 만족도 차트: 도넛/라인/막대]
    ↓
[질문 분석: Top 5 SOP, 질문 유형, 커버리지 히트맵]
    ↓
[피드백 리뷰 큐: 부정 피드백 검토 → 검토 완료 체크]
    ↓
[문서 추가 / Prompt 조정 / SOP 업데이트]
```

### 4.3 데이터 플로우

```
sop-qa-demo.html
    → POST /api/chat      → chat-log.json
    → POST /api/feedback  → feedback-log.json

admin-dashboard.html
    → GET  /api/dashboard → chat-log.json 집계
    → GET  /api/feedback  → feedback-log.json 집계
    → fallback            → feedback-demo.json (GitHub Pages)
```

## 5. v1 → v2 변경 사항 및 근거

### 5.1 v1 기능 범위

- SOP 10종 탑재 + BM25 기반 RAG 검색
- OpenAI API 연동 답변 생성
- 답변에 출처 섹션 표시
- 질문 로그(`chat-log.json`) 저장
- 관리자 대시보드 — Mock 데이터 기반 KPI 표시

### 5.2 v1 테스트 중 발견된 문제점

| # | 발견 문제 | 발견 방법 |
|---|-----------|-----------|
| 1 | 답변 품질 저하 여부를 감지할 피드백 수단이 없음 | QA 테스트 중 사용자 인터뷰 |
| 2 | 어떤 질문이 자주 실패하는지 가시화 불가 | 로그 수동 분석 |
| 3 | 대시보드가 Mock 데이터 고정 — 실운영 가치 없음 | 관리자 사용 후기 |
| 4 | 부정 답변에 대한 개선 루프 없음 | 서비스 운영 리뷰 |
| 5 | SOP 커버리지 편차 파악 불가 | 문서 기획팀 피드백 |

### 5.3 v2 변경 내용

- **피드백 루프 추가**: 답변마다 👍/👎 UI → 사유 선택 → 서버 저장
- **대시보드 고도화**: 실데이터 기반 만족도·질문 분석·커버리지 차트
- **피드백 리뷰 큐**: 부정 피드백 목록 + 검토 완료 체크
- **서비스 기획 문서화**: 운영 사이클 정착을 위한 문서 체계 수립

## 6. 핵심 지표 (KPI)

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 응답 만족도 (긍정 비율) | **≥ 80%** | feedback-log 긍정 수 / 전체 피드백 수 |
| SOP 커버리지율 | **≥ 70%** | 참조된 SOP 종수 / 전체 10종 |
| 평균 응답 시간 | **≤ 3초** | API 응답 시간 측정 |
| 부정 사유 Top 1 | 매월 변화 추이 관찰 | 불만족 사유 분포 차트 |
| 리뷰 큐 미처리 건수 | **≤ 3건 (상시)** | 피드백 리뷰 큐 미검토 수 |

## 7. 향후 로드맵 (v3)

### Phase 1 — 검색 품질 고도화 (단기)

- **멀티 SOP 비교 질문 지원**: "SOP-MFG-001과 SOP-QA-002의 세척 기준 차이는?"
- **후속 질문 자동 추천**: 답변 하단에 연관 질문 3개 제시
- **SOP 버전 관리 및 변경 알림**: 문서 업데이트 시 영향받는 질문 자동 감지

### Phase 2 — 운영 자동화 (중기)

- **이상 탐지 알림**: 만족도 급락·부정 피드백 급증 시 관리자 자동 알림
- **자동 CAPA 초안 생성**: 반복 부정 피드백 패턴 → 개선 초안 자동 생성
- **다국어 지원**: 영문 SOP 기반 한/영 혼합 질의 처리

### Phase 3 — 엔터프라이즈 확장 (장기)

- **EDMS 연동**: 승인 문서 자동 갱신 파이프라인
- **감사(Audit) 로그**: 규제기관 제출용 Q&A 이력 추적
- **Role-based 접근 제어**: 부서별 SOP 접근 권한 분리
